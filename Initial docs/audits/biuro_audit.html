Projekt na wysokim poziomie produkcyjnym

Autonomiczne Biuro to ambitna, dobrze zaprojektowana platforma orkiestracji agentów AI. Monorepo jest czysto podzielone na trzy pakiety (server, dashboard, cli), architektura heartbeat jest solidna i dobrze przetestowana, a stack bezpieczeństwa (RLS w PostgreSQL, RBAC, Zod validation, Helmet CSP) jest na poziomie enterprise. Projekt ma 73 testy — coverage gate wymaga ≥70% — i pełny pipeline E2E z Playwright. Główne luki: brak globalnego rate-limitingu REST API, kilka kluczowych serwisów wyłączonych z coverage, oraz AUTH_ENABLED=false w przykładzie dockera jako potencjalna pułapka dla początkujących.

✅ Mocne strony architektury

Monorepo pnpm z pakietami @biuro/server, @biuro/dashboard, @biuro/cli i wspólnym @biuro/shared. Heartbeat loop atomowy z FOR UPDATE SKIP LOCKED — brak race condition. Orchestrator podzielony na 11 modułów (heartbeat, budget, safety, actions, recording…). System migracji numerowanych z checksumami (v1–v14).

⚠️ Obszary do usprawnienia

Brak globalnego rate-limitingu na endpointach REST (tylko na auth). Worker i server w tym samym Dockerfile — lepiej oddzielne obrazy. Serwis collaboration.ts (627 linii) i nlCommandPlanner.ts (1208 linii) to kandydaci do rozbicia. Brak circuit-breaker przy wywołaniach LLM runtime.

🔐 Bezpieczeństwo

DOBRZE

Row Level Security w PostgreSQL

Schema v14 zawiera kompletne polityki RLS na wszystkich głównych tabelach (agents, tasks, goals, audit_log, company_knowledge i inne). Funkcje biuro_current_company_id() i biuro_is_internal_context() zapewniają izolację danych między firmami.

DOBRZE

Brak kluczy API w historii git

Przeszukano całe git log -p — żadnych kluczy sk-ant-* ani sk-*. Plik .env.example zawiera tylko placeholder'y. Template sanitizer automatycznie wyczysa wrażliwe pola przy eksporcie firm.

DOBRZE

Helmet, CSP, RBAC, Zod

Helmet z CSP headers, RBAC z 4 rolami (owner/admin/member/viewer) na poziomie firmy, walidacja wszystkich inputów przez Zod, hasła przechowywane jako hash (nie plaintext), klucze API hashed w bazie.

UWAGA

Brak globalnego rate-limitingu REST

Rate-limiting istnieje tylko na endpointach auth (AUTH_RATE_LIMIT_WINDOW_MS, AUTH_RATE_LIMIT_MAX) i WebSocket. Pozostałe 50+ endpointów REST jest bez ograniczeń — podatne na nadużycia lub ataki brute-force przy dłuższym czasie odpowiedzi LLM.

UWAGA

AUTH_ENABLED=false w docker-compose przykładzie

Komentarz w README i CI ustawia AUTH_ENABLED: 'true', ale domyślna wartość to ${AUTH_ENABLED:-true}. README wyraźnie ostrzega, ale nowi użytkownicy mogą łatwo pozostawić auth wyłączone w quickstart.

INFO

Bash sandbox z Docker

Bash tools uruchamiane w sandboxie Docker (alpine/git:2.47.2) z ograniczeniami pamięci (256MB), CPU (1 rdzeń), PID (64) i whitelistą binariów. WORKSPACE_ROOT oddzielony od systemu hosta.

⭐ Co wyróżnia projekt

🤖 NL Command Planner (1208 linii)

Serwis interpretacji poleceń naturalnego języka pozwala operatorowi pisać komendy jak "zatrudnij agenta CEO" zamiast klikalnia przez UI. Zwraca strukturyzowane akcje z Zod-walidowanym JSON. Imponująca złożoność domeny w jednym module.

🧠 Knowledge Base + Vector Search

Wbudowane embeddingi OpenAI (text-embedding-3-small), backfill embeddings dla istniejących dokumentów, RAG retrieval z metrykam trafności. Agenci mają dostęp do bazy wiedzy firmy.

💓 Heartbeat z pełnym telemetrią

Każdy heartbeat rejestruje: koszt tokenów, czas trwania, wynik, aktywny task. Osobne moduły dla budget, safety, actions, recording, telemetry. Prometheus eksportuje metryki, Grafana ma gotowy dashboard.

🏢 Multi-company z pełną izolacją

Jedno wdrożenie obsługuje wiele firm. RLS na poziomie PostgreSQL gwarantuje izolację danych. Każda firma ma własnych agentów, budżety, polityki, narzędzia i ścieżkę audytową.

📋 System szablonów

Eksport/import całych firm jako JSON templates (z automatycznym usuwaniem sekretów). Preset saas-startup.json tworzy pełną firmę z 5 agentami, celami i politykami w jednym poleceniu CLI.

🛡️ System bezpieczeństwa agentów

Ochrona przed runaway agents: rate-limiting heartbeatów (60/h), detekcja circular delegation, auto-pauza przy 5 błędach z rzędu, max 5 poziomów delegacji, max 100 wywołań narzędzi na task, max 24h na task.

🚨 Kluczowe problemy do rozwiązania

PRIORYTET 1

Globalny rate-limiting REST API

Dodać express-rate-limit jako middleware dla wszystkich routes (nie tylko auth). Sugerowane limity: 100 req/min dla standardowych endpointów, 20 req/min dla operacji LLM (create agent, trigger heartbeat). Biblioteka już jest w package.json, tylko nie zastosowana globalnie.

PRIORYTET 1

Testy dla billing i knowledge routes

Pliki billing.ts, knowledge.ts, approvals.ts, outgoingWebhooks.ts są wyłączone z coverage. To krytyczne obszary (płatności, webhook delivery) bez mierzalnego pokrycia testami. Należy albo napisać testy, albo uzasadnić wykluczenie.

PRIORYTET 2

Circuit-breaker dla wywołań LLM

Brak circuit-breakera przy wywołaniach do Claude/OpenAI/Gemini. Długotrwałe awarie LLM mogą wywołać kaskadowe blokady heartbeatów. Rozważyć bibliotekę opossum lub prosty licznik błędów z back-off.

PRIORYTET 3

nlCommandPlanner.ts wymaga refaktoringu

1208 linii to zbyt duży moduł — trudny do testowania jednostkowego. Rozważyć podział na: nlParser.ts (interpretacja), nlCommandRouter.ts (routing do akcji), nlResponseFormatter.ts (formatowanie odpowiedzi).

PRIORYTET 3

Template marketplace URL hardcoded

TEMPLATE_MARKETPLACE_URL=https://marketplace.biuro.ai/templates.json wskazuje na domenę, która może nie istnieć. Należy dodać fallback i lepszy error handling gdy marketplace jest niedostępny.
