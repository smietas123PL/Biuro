01
Live Agent Replay
WOW

Możliwość "przewinięcia" całej historii pracy dowolnego agenta — krok po kroku, jak debugger, ale dla AI. Każda decyzja, każde wywołanie narzędzia, każda myśl LLM widoczna w interaktywnej osi czasu z możliwością pauzy, przyspieszenia i skoku do dowolnego momentu.

To nie jest funkcja dla developerów — to narzędzie sprzedażowe i audytowe. Pokazanie klientowi "oto co twoi agenci zrobili przez ostatnie 8 godzin" jest nieporównanie bardziej przekonujące niż lista logów. Kluczowe przy onboardingu enterprise i podczas incydentów ("dlaczego agent X to zrobił?").

Dlaczego teraz?
Dane już istnieją. Tabele heartbeats, audit_log, messages i agent_sessions zawierają pełną historię. Brakuje jedynie warstwy prezentacji — odtwarzacza który przegląda rekordy w kolejności created_at i animuje stan.
Implementacja
Endpoint GET /api/agents/:id/replay?from=&to= zwraca posortowane eventy. Frontend: komponent TimelineScrubber z kontrolkami play/pause/speed. Każdy event renderuje się jako karta z ikoną, timestampem i kosztorem LLM.
Stack techniczny
React requestAnimationFrame PostgreSQL cursor Server-Sent Events
Efekt biznesowy
Skrócenie czasu debug sesji z godzin do minut. Materiał do demo dla enterprise. Podstawa pod compliance audit w regulowanych branżach (fintech, healthcare).
Nakład: 3–5 dni
Impact: bardzo wysoki
Ryzyko: niskie
02
Natural Language Control Panel
WOW

Jedno pole tekstowe w górnej belce dashboardu. Piszesz: "zatrzymaj wszystkich agentów i zrób snapshot stanu" — system rozumie, pyta o potwierdzenie i wykonuje. Albo: "przydziel 3 najstarsze taski agentowi Marta, priorytet wysoki". Zero nawigacji po menu, pełna kontrola przez język naturalny.

To Command Palette, ale napędzane Claude. Istniejący komponent CommandPalette.tsx jest już zaczątkiem — wystarczy podpiąć pod endpoint który tłumaczy free-text na sekwencję wywołań API. Dla użytkowników nietech to jedyna warstwa której potrzebują.

Architektura
POST /api/nl-command przyjmuje tekst → Claude z function calling mapuje na akcje REST → serwer zwraca plan do zatwierdzenia → user klika "Wykonaj" → akcje leciał sekwencyjnie z websocket feed postępu.
Bezpieczeństwo
Wszystkie destruktywne akcje (pause, terminate, delete) wymagają explicit confirmation step. Każda komenda ląduje w audit_log z source: "nl_command". Governance policies stosują się normalnie.
Przykładowe komendy
"pokaż agentów którzy przekroczyli budżet"
"utwórz taska z opisu maila który wklejam"
"jaki był koszt operacji w tym tygodniu?"
"cofnij ostatnią delegację do agenta HR"
Stack techniczny
Claude tool use WebSocket feedback CommandPalette.tsx Zod validation
Nakład: 5–8 dni
Impact: bardzo wysoki
Ryzyko: średnie
03
Autonomous Goal Decomposition
WOW

Wpisujesz cel wysokiego poziomu: "pozyskaj 50 leadów B2B z branży fintech w Q3". System rozbija go na hierarchię subtasków, przydziela każdy do odpowiedniego agenta na podstawie roli i historii, monitoruje postęp i automatycznie adaptuje plan gdy coś się nie powiedzie.

To moment w którym Autonomiczne Biuro przestaje być narzędziem do zarządzania agentami i staje się platformą do realizacji celów biznesowych. Struktura goals → tasks → subtasks już jest w schemacie bazy. Brakuje silnika który planuje w górę, nie tylko w dół.

Jak to działa
Goal Planner Agent (specjalny agent systemowy) otrzymuje cel i profil firmy. Generuje drzewo tasków z szacowanym kosztem i SLA. Manager Agent nadzoruje wykonanie, reaguje na blokady redelegacją i raportuje postęp do celu.
Adaptacja
Gdy task zostaje zablokowany lub przekracza budżet, Goal Planner jest wzywany ponownie z kontekstem niepowodzenia. Może wygenerować alternatywne podejście, eskalować do human approval lub oznaczyć cel jako "wymaga interwencji".
Zależności
Wymaga dojrzałego systemu delegacji (już jest), budżetów per goal (do dodania), i mechanizmu "goal health score" widocznego w dashboardzie. Powinno być zbudowane po NL Control Panel.
Stack techniczny
Goal Planner Agent Tree-of-Thoughts goal_budgets table GoalsPage.tsx
Nakład: 2–3 tygodnie
Impact: przełomowy
Ryzyko: wysokie
🎨 UI / UX
04
Real-time Activity Feed z kontekstem
UI/UX

Obecny dashboard pokazuje statyczne liczby (aktywni agenci: 3, taski: 47). Potrzebny jest żywy, chronologiczny feed "co dzieje się teraz" — z awatarem agenta, skrótem wykonywanego taska, typem akcji (tool call, delegation, memory store) i kosztem w czasie rzeczywistym.

WebSocket już jest w ws.ts i broadcast agent.working jest już emitowany przy heartbeacie. Brakuje komponentu który subskrybuje te eventy i renderuje je jako płynny stream bez full-page refresh. Użytkownik powinien widzieć biuro "oddychające" — żywe, nie zamrożone.

Komponenty do zbudowania
ActivityFeed.tsx — pionowy stream eventów z scroll-to-bottom
AgentPulse.tsx — pulsujący wskaźnik przy aktywnym agencie
CostTicker.tsx — licznik kosztu dnia aktualizowany co heartbeat
Grupowanie eventów
Eventy tego samego agenta z tego samego taska w oknie 30s grupują się w jeden wiersz rozwijany kliknięciem. Zapobiega zalewaniu feedu gdy agent działa intensywnie. Nagłówek grupy pokazuje "Agent X · 12 akcji · Task Y".
Filtrowanie
Chipy filtrów nad feedem: Wszystko / Błędy / Koszty / Delegacje / Tool calls. Persistent w localStorage. Brak filtru = widok dla managera. Tylko błędy = widok dla on-call.
Stack techniczny
useWebSocket hook React.memo IntersectionObserver ws.ts broadcast
Nakład: 2–3 dni
Impact: wysoki
Ryzyko: niskie
05
OrgChart jako centrum dowodzenia
UI/UX

Strona OrgChartPage.tsx istnieje, ale prawdopodobnie renderuje statyczne drzewo. Powinna być interaktywnym centrum dowodzenia: drag-and-drop przydzielania tasków między agentami, kolory węzłów kodujące obciążenie (zielony = idle, żółty = busy, czerwony = error/paused), sidepanel z detalami agenta po kliknięciu.

Dla managera zarządzającego 20+ agentami widok org-chartu jest jedyną nawigacją której potrzebuje. Chce jednym spojrzeniem ocenić czy system działa — bez przechodzenia między zakładkami Agents → Tasks → AgentDetail.

Drag & Drop
Przeciągnięcie taska (z listy po prawej) na węzeł agenta = przydzielenie. Przeciągnięcie agenta pod innego agenta = zmiana reports_to. Konfirmacja modalem z podglądem zmiany zanim trafi do API.
Status encoding
🟢 idle — gotowy na task
🟡 working — aktualnie przetwarza
🔴 paused — zablokowany, wymaga uwagi
⚫ terminated — nieaktywny
Pulsujący border = jest w trakcie heartbeatu
Sidepanel
Kliknięcie węzła = szuflada z prawej: aktualny task, ostatnie 5 heartbeatów, koszt miesięczny vs budżet, przycisk "Pause/Resume" i "Assign Task". Zero nawigacji na inną stronę.
Stack techniczny
d3-hierarchy @dnd-kit/core WebSocket live status Framer Motion
Nakład: 5–7 dni
Impact: wysoki

06
Spójny system błędów i powiadomień
UI/UX

Największa dziura UX w obecnym dashboardzie: brak ErrorBoundary, brak toast systemu, brak retry przy błędach sieciowych. Błędy API albo milczą (użytkownik nie wie że coś się nie zapisało) albo crashują cały komponent z białym ekranem.

W systemie gdzie agenci wykonują autonomiczne akcje za prawdziwe pieniądze, każdy błąd musi być widoczny i actionable. "Failed to pause agent" bez żadnego feedbacku to katastrofa UX — użytkownik klika trzy razy, wysyła zduplikowane requesty i nie wie czy akcja zadziałała.

Architektura powiadomień
Toast — sukces/info/warning (auto-dismiss 4s)
Inline error — przy polach formularza
Banner — degradacja serwisu, utrata WS
Full-page fallback — crash krytycznego komponentu
Smart retry
Każdy useApi hook dostaje retryCount: 3, backoff: exponential dla GET requestów. Przy 429/503 automatyczny retry z informacją "Ponawiam za 5s...". PUT/POST/DELETE bez retry — tylko confirm dialog przy błędzie.
Connection status
Pasek w nagłówku: zielona kropka = WS connected, szara = offline. Przy utracie połączenia banner "Połączenie przerwane — dane mogą być nieaktualne" z przyciskiem "Odśwież". Bez białych ekranów.
Stack techniczny
React ErrorBoundary Sonner / react-hot-toast useOnlineStatus hook axios interceptors
Nakład: 2–3 dni
Impact: wysoki
Ryzyko: niskie
🔧 Core — fundament produkcji
07
Pipeline migracji bazy danych
Core

12 plików schema_vN.sql bez automatycznego runnera który śledzi wersję w tabeli to bomba zegarowa. Każdy deployment jest zakładem — czy migrations zostały zastosowane w odpowiedniej kolejności, czy środowisko deweloperskie nie zdesynchonizowało się z produkcją?

Potrzebny runner który: przechowuje wersję schematu w tabeli schema_migrations, egzekwuje kolejność plików, blokuje start serwera gdy baza jest za stara, loguje każdą migrację z timestampem i checksumą. Wzorzec Flyway / Liquibase, zaimplementowany natywnie w TypeScript.

Tabela migrations
CREATE TABLE schema_migrations (
  version INT PRIMARY KEY,
  name TEXT NOT NULL,
  checksum TEXT,
  applied_at TIMESTAMPTZ DEFAULT now()
);
Runner logic
Skanuje /db/migrations/*.sql sortując po numerze prefiksu. Porównuje z tabelą. Stosuje brakujące w transakcji — rollback przy błędzie. Jeśli checksum istniejącej migracji się zmienił: fatal error, nie kontynuuje.
Integracja z serwerem
migrate.ts już istnieje. Trzeba wymusić jego uruchomienie w start() przed db.query('SELECT 1'). W docker-compose: osobny migrate service jako init container który musi skończyć przed startem server.
Stack techniczny
native TypeScript pg transactions schema_migrations table SHA256 checksum
Nakład: 1–2 dni
Impact: krytyczny
Ryzyko: niskie
08
Dokończyć integracje Slack / Discord
Core

IntegrationService.handleDiscordEvent zawiera komentarz "// Similar logic to Slack..." — to nieukończony stub. Slack i Discord to potencjalnie kluczowy kanał dla nietech użytkowników którzy chcą przydzielać taski i dostać raport bez otwierania dashboardu.

Dokończenie integracji oznacza: weryfikację podpisów przychodzących webhooków, pełny zestaw slash komend, dwukierunkową komunikację (agent może zapytać użytkownika o decyzję przez wiadomość), i status thread dla każdego aktywnego taska.

Slack slash commands
/biuro task [opis] — tworzy task
/biuro status [id] — status taska
/biuro agents — lista z obciążeniem
/biuro approve [id] — zatwierdza approval request
/biuro budget — dzienny koszt
Weryfikacja webhooków
SLACK_SIGNING_SECRET jest już w env schema — brakuje middleware który weryfikuje X-Slack-Signature przez HMAC SHA256. Bez tego endpoint jest publiczny dla każdego kto zgadnie URL.
Agent → Slack
Gdy agent tworzy approval_request, NotificationService wysyła wiadomość do kanału Slack z przyciskami Approve/Reject. Kliknięcie = webhook do API = rozwiązanie approval. Pętla zamknięta bez dashboardu.
Stack techniczny
@slack/bolt discord.js HMAC webhook verify Block Kit UI
Nakład: 4–6 dni
Impact: wysoki
Ryzyko: średnie
09
Row Level Security dla multi-tenancy
Core

Każda tabela ma company_id, ale izolacja odbywa się wyłącznie przez WHERE company_id = $1 w kodzie aplikacji. Jeden błąd w zapytaniu — pominięty warunek, zły parametr, copy-paste — i tenant A widzi dane tenanta B. To naruszenie które kończy firmę.

Row Level Security w PostgreSQL przenosi izolację na poziom bazy danych, niezależny od kodu aplikacji. Nawet jeśli developer popełni błąd w query, baza odmówi zwrócenia cudzych rekordów. To warstwa bezpieczeństwa której nie można przypadkowo wyłączyć.

Implementacja RLS
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON tasks
  USING (company_id = current_setting(
    'app.current_company_id')::uuid);
Powtórzone dla każdej tabeli z company_id.
Ustawianie kontekstu
Middleware Express po autentykacji: SET LOCAL app.current_company_id = '...' w każdej transakcji. Connection pool musi używać BEGIN/COMMIT żeby LOCAL scope działał poprawnie — przegląd użycia db.query kontra db.transaction.
Wyjątki
Tabele systemowe (schema_migrations, companies) bez RLS — dostęp przez superuser role. Agenty worker-a muszą mieć BYPASSRLS lub ustawiać company_id przed każdą operacją heartbeatu.
Stack techniczny
PostgreSQL RLS SET LOCAL connection pool review migration schema_v13
Nakład: 3–5 dni
Impact: krytyczny
Ryzyko: średnie
10
Observability — metryki i distributed tracing
Core

System loguje przez pino i ma audit_log, ale brakuje metryk i tracingu między heartbeat → LLM call → tool execution → db write. Gdy agent "utyka" lub nagle drożeje, nie wiadomo na którym etapie i dlaczego. Bez tego SLA dla klientów biznesowych jest nieosiągalne.

Observability to nie luksus — to warunek konieczny do prowadzenia systemu który autonomicznie wydaje pieniądze klientów. Musisz wiedzieć w czasie rzeczywistym: ile kosztuje każdy runtime, który tool call trwa najdłużej, gdzie są bottlenecki przy 20 równoległych heartbeatach.

Metryki Prometheus
biuro_heartbeats_total (by status, agent)
biuro_llm_cost_usd_total (by runtime, model)
biuro_llm_latency_ms (histogram)
biuro_tasks_completed_total
Endpoint GET /metrics dla Prometheus scrape.
Distributed tracing
OpenTelemetry trace od HTTP request przez processAgentHeartbeat → buildAgentContext → runtime.run → executeTool → db.query. Każdy span z atrybutami: agentId, taskId, model, cost. Eksport do Jaeger lub Honeycomb.
Alerty
Alertmanager reguły: agent error rate > 20% w oknie 5min, LLM p99 latency > 30s, dzienny koszt > 80% miesięcznego budżetu, brak heartbeatów przez > 2 minuty. Notyfikacja przez... Slack integration (punkt 08).
Stack techniczny
OpenTelemetry SDK prom-client Jaeger / Honeycomb Grafana dashboard
Nakład: 4–6 dni
Impact: wysoki
Ryzyko: niskie
