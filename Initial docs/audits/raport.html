Mocne strony projektu

Ambitna, spójna koncepcja

Projekt mapuje realne problemy orkiestracji agentów (budżety, hierarchie, heartbeaty) na dobrze przemyślany model domenowy. README jest wyjątkowo dobrej jakości — pełne diagramy, CLI, FAQ.

Solidny schemat bazy danych

Schema v1–v8 dobrze modeluje kompanie, agentów, zadania, polityki i audit_log. UUID PK, ON DELETE CASCADE, CHECK constraints, UNIQUE — poprawne wzorce PostgreSQL.

Atomowy checkout zadań

FOR UPDATE SKIP LOCKED w heartbeat.ts to prawidłowe rozwiązanie problemu wyścigu przy równoległych agentach. To najlepsza techniczna decyzja w projekcie.

Modułowy system runtime'ów

Interfejs IAgentRuntime z registry pozwala dodawać nowe LLM bez modyfikacji rdzenia. Claude, OpenAI, Gemini — dobra abstrakcja strategia design pattern.

Safety system w heartbeat

Detekcja pętli delegacji, flood wiadomości, auto-pause — skoordynowane bezpieczeństwo operacyjne, rzadkie w projektach na tym etapie.

Monorepo + Docker Compose

pnpm workspaces + docker-compose z osobnym workerem i dashboardem to dojrzała struktura deploymentu dla projektu open-source.

Problemy krytyczne (bezpieczeństwo)

natychmiastowe działanie
KRYTYCZNE
Hasła hashowane jako "hash_<password>" — BRAK bcrypt
packages/server/src/routes/auth.ts · linia ~20

Hasła są „hashowane” przez konkatenację z prefixem. To plaintext storage — każdy z dostępem do bazy może odczytać wszystkie hasła. Absolutnie niedopuszczalne w jakimkolwiek deploymencie.

// OBECNY KOD — KRYTYCZNA PODATNOŚĆ const pwdHash = `hash_${password}`; // POPRAWKA — użyj bcrypt import bcrypt from 'bcrypt'; const pwdHash = await bcrypt.hash(password, 12); // Weryfikacja: const valid = await bcrypt.compare(password, user.pwd_hash);
Wymagane zmiany:

Zainstaluj bcrypt lub argon2. Usuń komentarz „In a real app, use bcrypt” i wdróż teraz. Wszystkie obecne konta mają niezabezpieczone hasła.

KRYTYCZNE
AUTH_ENABLED=false domyślnie — brak autentykacji w produkcji
.env.example · docker-compose.yml

Domyślna konfiguracja całkowicie pomija autentykację. Każdy, kto zna URL API, może tworzyć agentów, uruchamiać narzędzia i wydawać środki. Docker Compose nie ustawia AUTH_ENABLED=true.

# docker-compose.yml — brak AUTH_ENABLED=true server: environment: DATABASE_URL: ... # AUTH_ENABLED nie ustawione → domyślnie false # POPRAWKA AUTH_ENABLED: "true"
Wymagane zmiany:

Ustaw AUTH_ENABLED=true w docker-compose. Opcja false powinna być tylko dla local dev — zaktualizuj README.

KRYTYCZNE
Brak walidacji danych z LLM przed wykonaniem narzędzi (prompt injection)
packages/server/src/orchestrator/heartbeat.ts

Akcje z odpowiedzi LLM (action.tool_name, action.params) nie są walidowane schematem przed wykonaniem. Złośliwy lub manipulowany model może zwrócić dowolną nazwę narzędzia lub parametry, które trafią do executora.

// RYZYKO: action pochodzi z LLM, może być manipulowany const result = await executeTool(agentId, task.id, action.tool_name, // brak walidacji action.params // brak walidacji ); // POPRAWKA: waliduj przez zod przed wykonaniem const ActionSchema = z.object({ tool_name: z.string().regex(/^[a-z_]+$/), params: z.record(z.unknown()) }); const safe = ActionSchema.parse(action);

Wysokie priorytety (architektura i stabilność)

przed alpha release
WYSOKIE
db.transaction() prawdopodobnie nie istnieje w client.ts — atomowy checkout niepewny
packages/server/src/db/client.ts · heartbeat.ts linia ~25

Heartbeat wywołuje db.transaction(async (client) => ...), ale standardowy pg.Pool tej metody nie posiada. Jeśli client.ts jej nie implementuje, atomowy checkout zadań w ogóle nie działa — dwóch agentów może pobrać to samo zadanie.

// Wymagana implementacja w packages/server/src/db/client.ts: export const db = { query: pool.query.bind(pool), transaction: async (fn) => { const client = await pool.connect(); try { await client.query('BEGIN'); const result = await fn(client); await client.query('COMMIT'); return result; } catch (e) { await client.query('ROLLBACK'); throw e; } finally { client.release(); } } };
WYSOKIE
Heartbeat nie zmienia statusu agenta na 'working' — wielokrotne równoległe uruchomienia
packages/server/src/orchestrator/heartbeat.ts · scheduler.ts

Scheduler pobiera tylko agentów ze statusem 'idle'. Ale heartbeat nigdzie nie ustawia agenta na 'working'. Oznacza to, że w każdym interwale (30s) ten sam agent może być uruchomiony wielokrotnie równolegle.

// Dodać na początku heartbeat: await db.query("UPDATE agents SET status='working' WHERE id=$1", [agentId]); // ...i w bloku finally: await db.query("UPDATE agents SET status='idle' WHERE id=$1", [agentId]);
WYSOKIE
8 plików schema_v*.sql bez systemu śledzenia migracji
packages/server/src/db/schema.sql – schema_v8.sql

Schema ewoluowała przez 8 addytywnych plików SQL. Plik migrate.ts istnieje, ale bez tabeli śledzenia wersji użytkownik musi uruchamiać każdy plik ręcznie i w odpowiedniej kolejności. To niemożliwe do utrzymania.

// migrate.ts powinien implementować: // 1. CREATE TABLE IF NOT EXISTS schema_migrations (version INT PRIMARY KEY) // 2. Sprawdź które v1..v8 nie zostały zastosowane // 3. Uruchom brakujące w kolejności // 4. Zapisz wersję // Alternatywa: użyj drizzle-kit, prisma migrate lub db-migrate
WYSOKIE
Claude runtime: hardkodowany przestarzały model claude-3-5-sonnet-20240620
packages/server/src/runtime/claude.ts · linia ~50

Model z połowy 2024 zamiast aktualnego. Schema ma kolumnę model w tabeli agents, ale runtime ją całkowicie ignoruje — to też oznacza, że per-agent konfiguracja modelu nie działa.

model: 'claude-3-5-sonnet-20240620', // POPRAWKA: użyj modelu z kontekstu agenta model: context.agent_model ?? 'claude-sonnet-4-6',

Dashboard: DashboardPage z fikcyjnymi danymi, większość stron to Placeholder
packages/dashboard/src/App.tsx

Strona główna wyświetla hardkodowane liczby (12 agentów, 45 tasków, $12.42). Strony Goals, Tools i inne to puste komponenty Placeholder. Rozbieżność między obszernym README a rzeczywistym stanem UI jest istotna dla projektu open-source.

<Route path="goals" element={<Placeholder title="Goals" />} /> <p className="text-2xl font-bold">12</p> // zawsze "12 agentów" // Wymagane: useApi hook + /api/companies/:id/stats const { data } = useApi('/companies/' + companyId + '/stats'); <p>{data?.active_agents ?? '...'}</p>

docker-compose nie przekazuje kluczy API do server/worker — agenci nie będą działać
docker-compose.yml

Server container ma tylko GOOGLE_API_KEY. Brak ANTHROPIC_API_KEY i OPENAI_API_KEY. Claude runtime falluje na mock, OpenAI zwróci 401. Worker też nie ma kluczy.

server: environment: ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY} OPENAI_API_KEY: ${OPENAI_API_KEY} GOOGLE_API_KEY: ${GOOGLE_API_KEY}

ŚREDNIE
Safety: limity z README (60/h heartbeat, 100 tools/task, 24h) nie są zaimplementowane
packages/server/src/orchestrator/safety.ts

README dokumentuje 9 mechanizmów ochrony. Kod implementuje tylko 3: circular delegation, message flood i consecutive errors. Brakuje: rate limit heartbeat, limit tool calls per task, max task duration, budget per task.

ŚREDNIE
Brak walidacji Zod w większości tras REST
packages/server/src/routes/

Trasy auth mają Zod (SignupSchema), ale agents, tasks, goals, policies — większość nie waliduje body. Brak walidacji to ryzyko błędów runtime i zniekształconych danych w bazie.

ŚREDNIE
Parsowanie odpowiedzi LLM przez regex — kruche
packages/server/src/runtime/claude.ts linia ~60

Odpowiedź agenta parsowana przez regex. Każda zmiana formatu lub nieoczekiwany znak w JSON powoduje cichą awarię (puste actions). Lepsza jest ekstrakcja przez Anthropic tool_use (JSON schema).

const actionsMatch = text.match(/<actions>([\s\S]*?)<\/actions>/); // Alternatywa: Anthropic tool_use wymusza strukturę tools: [{ name: "act", input_schema: { type: "object", ... } }]
ŚREDNIE
Brak restart policy i healthcheck w docker-compose
docker-compose.yml

Server i worker nie mają restart: unless-stopped ani healthcheck. Crash agenta nie jest automatycznie naprawiany. Endpoint /api/health istnieje w README, ale nie jest używany przez Docker.

server: restart: unless-stopped healthcheck: test: ["CMD", "curl", "-f", "http://localhost:3100/api/health"] interval: 30s retries: 3

Niskie priorytety i dobre praktyki

NISKIE
Brak indeksów na często odpytywanych kolumnach — seq scan przy >1k rekordów
packages/server/src/db/schema.sql

Heartbeat odpytuje tasks WHERE assigned_to AND status IN (...) i heartbeats WHERE agent_id AND created_at bez indeksów. Przy 1000+ rekordach każdy heartbeat (co 30s) będzie pełny scan.

CREATE INDEX idx_tasks_agent_status ON tasks(assigned_to, status); CREATE INDEX idx_heartbeats_agent_time ON heartbeats(agent_id, created_at DESC); CREATE INDEX idx_audit_company ON audit_log(company_id, created_at DESC); CREATE INDEX idx_messages_task_agent ON messages(task_id, from_agent, created_at);
NISKIE
7 plików verify*.ts w src/ — dług techniczny zaciemniający strukturę
packages/server/src/verify1b.ts — verify5_intelligence.ts

Prawdopodobnie skrypty weryfikacji faz development. Nie opisane w README, nie mają testów, zaciemniają strukturę projektu. Należy przenieść do scripts/ lub usunąć przed pierwszym publicznym release.
