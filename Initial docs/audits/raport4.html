CRITICAL WebSocket — zero autoryzacji, eventy firmy dostępne publicznie po UUID ws.ts · niezmieniony od v1

Przez 4 raporty i 4 sprinty poprawek ws.ts nie był dotknięty. Każdy kto zna UUID firmy — który pojawia się w URL dashboardu, nagłówkach API, logach — może nawiązać połączenie WebSocket i odbierać wszystkie live eventy: agent.working, heartbeat.completed, approval.requested. Te eventy zawierają tytuły tasków, myśli agentów i koszty operacji.

ws.ts — identyczny od commitu v1: // Brak: token = url.searchParams.get('token') // Brak: const session = await getSession(token) // Brak: if (!session) { ws.close(4401, 'Unauthorized'); return; } const companyId = url.searchParams.get('companyId'); if (!companyId) { ws.close(); return; } this.clients.get(companyId)!.add(ws); // ← przyjmuje każdego
Poprawka: Dodać do URL parametr token, zaimportować getSession() z middleware/auth.ts, sprawdzić user_roles dla companyId. Odrzucić przez ws.close(4401). Frontend: dodać token do URL w useWebSocket(). Czas: ~2–3 godziny.
CRITICAL gemini-3.1-flash-lite nie istnieje w Gemini API — wszystkie agenty Gemini bez custom model crashują runtime/defaultModels.ts

Nowy plik defaultModels.ts definiuje gemini: 'gemini-3.1-flash-lite' — model który nie istnieje. Gemini API Google nie ma wersji 3.1 (najnowsza to 2.5 Flash / 2.0 Flash). Każdy heartbeat agenta z runtime='gemini' i bez ustawionego agent_model zakończy się błędem 404 Model not found rzuconym przez SDK, co przejdzie przez catch i zostanie zalogowane jako error heartbeatu — agent zostanie auto-pauzowany przez safety system po 5 kolejnych błędach.

runtime/defaultModels.ts — nieistniejący model: gemini: env.GEMINI_MODEL || 'gemini-3.1-flash-lite', // Poprawka — jeden z istniejących modeli: gemini: env.GEMINI_MODEL || 'gemini-2.0-flash', // lub: 'gemini-2.5-flash-preview-05-20' (jeśli dostępny)
Poprawka (1 minuta): Zmienić na 'gemini-2.0-flash' lub 'gemini-2.5-flash'. Sprawdzić listę dostępnych modeli: https://ai.google.dev/gemini-api/docs/models.
HIGH Budget race condition — SELECT + UPDATE w dwóch niezależnych operacjach orchestrator/heartbeat.ts:28–35

Graceful shutdown naprawiony, ale race condition budżetu nadal istnieje. Sprawdzenie limitu i aktualizacja wydatków to dwie oddzielne operacje. Przy równoległych heartbeatach (co scheduler uruchamia dla wszystkich idle agentów jednocześnie) dwa procesy mogą przejść sprawdzenie w tym samym momencie — zanim którykolwiek zaktualizuje saldo.

Poprawka atomowa: Zastąpić SELECT + UPDATE jednym zapytaniem: UPDATE budgets SET spent_usd = spent_usd + $2 WHERE agent_id = $1 AND month = date_trunc('month', now())::date AND spent_usd + $2 <= limit_usd RETURNING spent_usd. Jeśli 0 wierszy — budżet przekroczony. Jedno round-trip, zero race condition.
HIGH Docker — hasło bazy danych "password" hardcoded przez wszystkie 4 raporty docker-compose.yml

Czwarty raport z rzędu. POSTGRES_PASSWORD: password i postgres://biuro:password@db w trzech miejscach pliku docker-compose. Każdy kto ma dostęp do repozytorium ma dostęp do produkcyjnej bazy. Jedna linia zmiany.

Poprawka: POSTGRES_PASSWORD: ${DB_PASSWORD:?DB_PASSWORD must be set} — operator :? wymusi błąd przy braku zmiennej. Dodać DB_PASSWORD=CHANGE_ME_$(openssl rand -hex 12) do .env.example z komentarzem.

ANALIZA
DOBRY Hybrid RAG — deterministyczny fallback to przemyślane rozwiązanie services/embeddings.ts

Zamiast prostego "brak klucza API = crash", system wraca na deterministyczny hash-based embedding: tokenizacja, FNV hash, ważone wektory, normalizacja L2. To nie jest semantyczne wyszukiwanie, ale jest powtarzalne — to samo zdanie zawsze daje ten sam wektor. Hybrid search łączy wyniki wektorowe z lexical ILIKE scoring. Metryki retrieval zapisywane do retrieval_metrics tabeli.

Uwaga: ivfflat index z lists=100 wymaga minimum ~1000–3000 wektorów dla sensownych wyników. Na pustej bazie PostgreSQL i tak wykona sequential scan (poprawnie), ale ostrzeżenie przy tworzeniu indeksu jest warto dodać do dokumentacji.

DOBRY Template service — pierwsza kompletna implementacja z dry-run preview services/template.ts · ~800 linii

Największy plik w projekcie. Export pokrywa teraz agentów z reports_to_ref, tools z przypisaniami, policies, goals z hierarchią parent_ref, budgets. Import z mapowaniem ref → nowy UUID. Dry-run zwraca dokładną listę zmian przed zastosowaniem. Sanityzacja secretów przez pattern matching na kluczach. Trzy presets: solo-founder, product-delivery i inne.

Uwaga do poprawy: Import agenta z nieznanym runtime wartością przejdzie przez Zod (enum jest ['claude','openai','gemini']), ale co jeśli ktoś wyeksportuje szablon z przyszłej wersji z nowym runtimem? Brak graceful degradation dla nieznanych runtimeów.

DOBRY BudgetsPage — kompletna implementacja z prognozą i 7-dniowymi wykresami pages/BudgetsPage.tsx · utils/budgets.ts

Logika budżetowa wydzielona do utils/budgets.ts z czystymi funkcjami pure (buildDailySpendSeries, buildBudgetForecast, summarizeAgentBudgets). Właśnie te funkcje mają testy w budgets.test.ts. Frontend pokazuje progress bary per agent, alerty przy przekroczeniu, sparkline spending. Endpoint /companies/:id/budgets-summary zwraca wszystko w jednym wywołaniu.

UWAGA Testy — tylko unit na pure functions, zero testów DB lub orchestratora tests/*.test.ts

6 plików testowych to ogromny postęp, ale pokrycie jest selektywne. Testy pokrywają tylko kod bez side-effectów: buildDailySpendSeries(), buildDeterministicEmbedding(), template schema validation. Zero testów dla kodu który faktycznie może się posypać w produkcji:

  • Brak testów heartbeat — atomic checkout, budget enforcement, action processing
  • Brak testów safety.ts — circular delegation detection, flood detection
  • Brak testów graceful shutdown — czy taski są poprawnie odblokowywane?
  • Brak testów auth middleware — token validation, role checking, RLS context

Vitest config nie ma testcontainers ani pg-mem — żaden test nie może dotknąć bazy danych.

Następny krok: Dodać pg-mem jako in-memory PostgreSQL dla testów jednostkowych z zapytaniami. Nie wymaga Dockera w CI. Zacząć od safety.test.ts — detectCircularDelegation jest algorytmem który można przetestować z mock DB bez kontenera.
UWAGA Graceful shutdown — stopOrchestrator() nie czeka na aktywne heartbeaty worker.ts · orchestrator/scheduler.ts

Graceful shutdown jest zdecydowanie lepszy niż brak obsługi SIGTERM, ale ma lukę: stopOrchestrator() woła clearInterval(interval) — co zapobiega nowym heartbeatom, ale nie czeka na zakończenie aktywnych. Heartbeaty uruchomione przed SIGTERM nadal działają w tle, a db.close() jest wywoływany zaraz po. Aktywny heartbeat który trafi na zamknięte połączenie z bazą rzuci błąd i task znów zostanie zablokowany.

Poprawka: Dodać licznik aktywnych heartbeatów: let activeHeartbeats = 0. Na początku każdego processAgentHeartbeat: activeHeartbeats++, w finally: activeHeartbeats--. W shutdown: while (activeHeartbeats > 0) await sleep(200) przed db.close(). Lub alternatywnie: po stopOrchestrator wykonać UPDATE tasks SET status='assigned', locked_by=NULL WHERE locked_by IS NOT NULL.

PROJEKT JEST GOTOWY NA BETA — Z JEDNYM WARUNKIEM

Cztery iteracje, cztery raporty, każdy sprint zamykający kilkanaście problemów. Ewolucja jest wyraźna i imponująca. Commit 1180762 — 50 plików, 6400+ linii — to największy skok jakościowy: prawdziwy RAG z deterministycznym fallbackiem, kompletny system szablonów, testy jednostkowe, BudgetsPage z prognozami, Command Palette, graceful shutdown, live activity feed.

Projekt przeszedł od "niebezpiecznego prototypu" do "solidnej platformy beta" w czterech tygodniach iteracji.

Jeden warunek blokujący: WebSocket bez autoryzacji. Jeśli projekt ma być dostępny publicznie, każdy z UUID firmy może podsłuchać wszystkie operacje w czasie rzeczywistym. To 2–3 godziny pracy i jedyna rzecz stojąca między aktualnym kodem a bezpiecznym publicznym beta.

Jeden krytyczny bug: gemini-3.1-flash-lite nie istnieje w API Google — poprawka to jedna minuta. Nienaprawiony, crash każdego agenta Gemini bez custom modelu.

Po zamknięciu tych dwóch rzeczy: backend API jest w ~95% gotowości, frontend w ~85%, testy w ~45% (unit) / 0% (integration). Główny dług techniczny to testy integracyjne — safety.ts, heartbeat.ts i context.ts mają rosnącą złożoność bez żadnej siatki regresji.

Fundamenty są produkcyjnej jakości. Hybrid RAG, kompletny template system, multi-tenant RLS, HMAC integrations, scrypt auth, Command Palette — to nie jest prototyp. To platforma.
